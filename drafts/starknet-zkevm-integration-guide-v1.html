Stack Blueprint

The zkEVM stack on StarkNet layers a Solidity–compatible bytecode interpreter on top of Cairo’s STARK–proven execution model. First, the starknet–evm–compiler translates Solidity IR into Cairo assembly, preserving the EVM’s 256–bit word semantics. The generated Cairo program is then packaged as a StarkNet contract with the @external entry point _execute. At runtime the STWO prover ingests the raw calldata, executes the Cairo code, and emits a succinct proof that the EVM state transition is valid.

From an API perspective the flow looks like this:

POST /v1/compile
  { "source": "contract.sol", "optimizer": true }

POST /v1/prove
  { "cairoartifact": "0xabc...", "calldata": "0x1234" }

POST /v1/preconfirm
  { "proof": "0xdef...", "txhash": "0x4567" }

The /v1/prove endpoint streams the proof in chunks (≈ 2 MiB for a typical 100–instruction transaction) and returns a proofid that the preconfirmation service can poll for readiness.

Integrating this stack into an existing CI pipeline involves three steps: (1) add the compiler as a Docker image (starknet/evm-compiler:latest), (2) configure a webhook that forwards compiled artifacts to the STWO prover, and (3) wrap the /v1/preconfirm call in a retry loop that respects the 1–second latency SLA. The following Rust pseudo–code illustrates the retry logic:

let mut attempts = 0;
loop {
    let res = client.post("/v1/preconfirm", &payload).await?;
    if res.status().issuccess() { break; }
    attempts += 1;
    if attempts > 5 { panic!("Preconfirmation failed"); }
    tokio::time::sleep(Duration::frommillis(200)).await;
}

Performance Trade–offs

Embedding a zkEVM adds two measurable overheads: proof generation cost and data–availability (DA) payload size. The STWO prover currently charges a fee per proof, translating to a specific cost at today’s price. In contrast, a plain Cairo transaction without proof costs roughly a fraction of that. The DA payload, stored in EIP–4844 blobs, grows by a certain percentage because each EVM opcode is serialized into a STARK trace.

Benchmarks from recent rollouts show a significant reduction in gas–equivalent consumption for a typical ERC–20 transfer, while preserving EVM compatibility at the bytecode level.

When designing a high–throughput DApp you must decide how many calls to batch into a single proof. Empirically, bundling up to 50 EVM calls yields a proof size of ≈ 1 MiB and keeps the prover latency under 0.9 s. Beyond that the marginal cost curve steepens, and the preconfirmation service begins to reject bundles that would breach the 1–second latency SLA.

Consensus & Integration

StarkNet’s consensus layer is a proof–of–stake system where validators submit STWO proofs as part of block finalization. To register a zkEVM contract you must stake the minimum amount and signal the feature flag via the governance contract. The flag unlocks the preconfirmation hook, which routes incoming proofs to the “zkEVM” shard of the validator set.

From a developer standpoint the only on–chain change is the addition of a struct that records the prover’s public key. The following Cairo snippet demonstrates the registration call:

@external
func registerzkevmvalidator{syscallptr: felt*}(
    proverkey: felt
) -> ():
    let (validator) = getvalidator(msg.sender)
    validator.proverkey = proverkey
    return ()
end

Because the zkEVM shares the same fee token, you can leverage existing gas–price oracle contracts without modification. However, the fee calculation now includes a term that the runtime extracts from the proof metadata. Updating the calculatefee function ensures validators are reimbursed for the extra CPU cycles.

Scenario: Scaling a DEX to $100M TVL

Imagine you are porting a Uniswap–v3–style DEX onto StarkNet’s zkEVM. The target is $100 M TVL within six months, requiring ~150 k swaps per day. By batching swaps per proof you generate roughly 6 k proofs daily. At a certain cost per proof the additional cost is a fraction of the liquidity–provider fees you’d earn at a specific swap fee.

Implementation steps: (1) replace the Solidity router with a Cairo–generated wrapper that exposes a batched function, (2) configure the backend to collect pending swap calldata into a buffer, (3) invoke the /v1/prove endpoint once per buffer, and (4) submit the resulting proof to the preconfirmation gateway. Monitoring tools should watch the prooflatencyms metric; staying under 900 ms guarantees the DEX remains responsive for UI–driven trades.

Post–launch, the DEX can tap into staking rewards to subsidize user fees. By allocating a certain percentage of the TVL to a liquidity mining pool that distributes a specific APY, the protocol aligns validator incentives with liquidity providers, closing the loop between consensus participation and on–chain economic activity.
